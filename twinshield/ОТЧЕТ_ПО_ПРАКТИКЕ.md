## 1. ОПИСАНИЕ ПРОЕКТА

### 1.1 Общая информация

**Egiz** — это инновационная система поведенческой идентификации для защиты от несанкционированного доступа. Проект представляет собой полнофункциональный MVP (Minimum Viable Product) для хакатона, который позволяет аутентифицировать пользователей на основе анализа их поведения при взаимодействии с устройством.

**Ключевая концепция:** *"Ваше поведение — это ключ. Всё остальное — просто предположение"*

### 1.2 Цели проекта

- Разработать систему поведенческой аутентификации как альтернативу традиционным методам (пароли, двухфакторная аутентификация)
- Создать мобильное приложение для сбора и анализа данных о поведении пользователя
- Разработать бэкенд-сервис для обработки и оценки поведенческих данных
- Реализовать адаптивную систему многофакторной аутентификации на основе риска
- Демонстрировать возможности ИИ для объяснения решений о риске

## 3. ФУНКЦИОНАЛЬНЫЕ ВОЗМОЖНОСТИ

### 3.1 Основной процесс работы

1. **Регистрация** → Создание аккаунта с хешированием пароля (PBKDF2-SHA256)
2. **Обучение** → Сбор 3-5 образцов поведения (набор текста, касания, свайпы)
3. **Проверка** → Сравнение новой сессии с базовым профилем поведения
4. **Действие** → Оценка риска и рекомендация (Разрешить → MFA)

### 3.2 Реализованные особенности (v0.3)

| Функция | Описание |
|---------|---------|
| **Оценка риска + объяснение ИИ** | Система выдает показатель риска (0-100) с понятным объяснением |
| **Адаптивная MFA** | Трёхуровневая система (none/soft/hard) в зависимости от риска |
| **Доверенные устройства** | Регистрация текущего устройства для снижения подозрений |
| **Центр угроз** | Логирование и визуализация недавних проверок с live-статусом |
| **Невозможный путь** | Детектор перемещений (новое местоположение в пределах 2ч) |
| **Детектор эмулятора** | Определение работы в симуляторе (expo-device) |
| **Визуализация "цифрового двойника"** | Графики сравнения (Δ%) параметров: набор текста, касания, свайпы |
| **Имитация хакера** | One-tap пресет для демонстрации атаки на экране проверки |

### 3.3 API Endpoints

| Метод | Путь | Описание |
|-------|------|---------|
| GET | `/health` | Статус сервиса |
| POST | `/register` | Регистрация нового пользователя |
| POST | `/login` | Вход в систему |
| GET | `/users/{id}/profile` | Профиль пользователя (образцы, базовая линия, доверенные устройства) |
| POST | `/users/{id}/trusted-devices` | Добавить доверенное устройство |
| DELETE | `/users/{id}/trusted-devices?device=...` | Удалить доверенное устройство |
| POST | `/users/{id}/heartbeat` | Live-пинг для Dashboard |
| GET | `/users/{id}/threat-dashboard` | Журнал угроз + статистика + live-сессия |
| POST | `/train` | Сохранить образец поведения |
| POST | `/check` | Полная проверка с оценкой риска и объяснением |

## 4. РЕАЛИЗАЦИЯ BACKEND

### 4.1 Основные компоненты

#### **Безопасность паролей**
```python
def hash_password(password: str) -> str:
    salt = secrets.token_hex(16)
    digest = hashlib.pbkdf2_hmac("sha256", password.encode(), 
                                  salt.encode(), 120_000)
    return f"{salt}${digest.hex()}"
```

- Используется PBKDF2-SHA256 с 120,000 итерациями
- Случайная соль для каждого пароля
- Timing-safe сравнение паролей (secrets.compare_digest)

#### **Схема БД**
- `users` — учетные записи пользователей
- `behavior_samples` — образцы поведения
- `trusted_devices` — доверенные устройства
- `threat_log` — журнал угроз и проверок
- `live_sessions` — текущие активные сессии

#### **Алгоритм оценки риска**

**Ключевые параметры:**
- Минимум образцов для начала проверки: 3
- Окно невозможного путешествия: 7200 сек (2 часа)
- Штраф за невозможное путешествие: 35 баллов
- Штраф за эмулятор: 18 баллов

**Расчет риска:**
1. Сравнение параметров поведения (скорость набора, касания, свайпы)
2. Анализ девиации от базовой линии
3. Проверка устройства и геолокации
4. Вычисление итогового Score (0-100)

### 4.2 CORS и безопасность

Настроена политика CORS для разработки:
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

⚠️ **Примечание:** В production необходимо ограничить `allow_origins` конкретными доменами.

### 4.3 Миграции БД

Автоматическое создание таблиц при запуске:
- Функция `init_db()` для первичной инициализации
- Функция `migrate()` для дополнительных таблиц

## 6. КЛЮЧЕВЫЕ ТЕХНИЧЕСКИЕ РЕШЕНИЯ

### 6.1 Сетевое взаимодействие

**Mobile → Backend:**
- Телефон и ПК должны быть в **одной сети** (Wi-Fi или мобильный хотспот)
- Использование локального IPv4 адреса (не 127.0.0.1)
- Конфигурация в `mobile/src/config.ts`

**Проблема портов:**
```powershell
netstat -ano | findstr ":8000"
taskkill /PID <PID> /F
```

### 6.2 Детектор эмулятора

Используется библиотека `expo-device` для определения работы в эмуляторе:

### 6.3 Хранение данных на устройстве

## 8. ДЕМОНСТРАЦИОННЫЙ СЦЕНАРИЙ

Для показа жюри хакатона подготовлен следующий сценарий:

| Шаг | Действие | Результат |
|-----|---------|-----------|
| 1 | Регистрация | Новый пользователь, сессия сохранена на устройстве |
| 2 | Обучение (Training) | Сбор 3-5 нормальных сессий → заполнение прогресс-бара |
| 3 | Проверка (нормальная) | Low risk → **Разрешить** (Allow) |
| 4 | Проверка (атака) | Быстрый набор, много касаний, неизвестное устройство, неизвестное место → **Высокий риск**, требование MFA, разбор факторов |

## 9. РУКОВОДСТВО ПО ЗАПУСКУ

### 9.1 Backend

```bash
cd backend
python -m venv venv
.\venv\Scripts\activate  # Windows
source venv/bin/activate  # Linux/Mac
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --reload --port 8000
```

Swagger UI: http://127.0.0.1:8000/docs

### 9.2 Mobile

1. Получить IPv4 адрес ПК: `ipconfig` → Wi-Fi адрес
2. Обновить `mobile/src/config.ts`: `API_URL = "http://YOUR_IP:8000"`
3. Убедиться в общей сети (Wi-Fi или хотспот)
4. Добавить правило firewall (Windows Admin):
```powershell
netsh advfirewall firewall add rule name="Egiz API" dir=in action=allow protocol=TCP localport=8000
```
5. Запустить:
```bash
cd mobile
npm install
npx expo start -c
```
6. Сканировать QR код в **Expo Go** (iOS/Android)

## 10. ОСНОВНЫЕ ВЫЗОВЫ И РЕШЕНИЯ

### Вызов 1: Сетевое взаимодействие мобилка ↔ ПК

**Проблема:** Устройства на разных сетях не могут взаимодействовать  
**Решение:** Требование единой сети (Wi-Fi или мобильный хотспот)

### Вызов 2: Эмулятор vs реальное устройство

**Проблема:** Разные поведенческие паттерны на эмуляторе  
**Решение:** Использование `expo-device` для детектирования и штрафного балла

### Вызов 3: Сохранение состояния при перезагрузке приложения

**Проблема:** Потеря данных о сессиях  
**Решение:** Async Storage для персистентности

### Вызов 4: Объяснение решений ИИ

**Проблема:** "Чёрный ящик" алгоритма риска  
**Решение:** Встроенное объяснение с разбором каждого фактора риска
